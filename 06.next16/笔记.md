# Linking and Navigating

## Navigating 怎么工作的？

1. Server Rendering
   App Router默认都是服务端组件，路由对应的 UI 树在服务端由 React/Next 计算出来，再把结果交给客户端展示：
   分为两类：

- Prerendering（预渲染）：构建时，或在 revalidation（重新验证） 触发时；结果可被 缓存。
- Dynamic Rendering（动态渲染）：请求时，响应某次客户端导航/请求。更「实时」，但更容易让用户感到「在等服务器」。
  服务端渲染 的代价是「新路由要等服务器算完->回应」；Next.js 用 Link （prefetch） + 客户端过渡（client-side transition） 把「等待」尽量挪到点击之前，并让「点到之后」不换整页、不重挂整棵布局， 体感更像 SPA
  维度 | Pages Router | App Router |
  |------|--------------|------------|
  | 默认组件模型 | `pages/*.tsx` 页面组件偏「页面级 CSR/SSR 混合」经典模型（`getServerSideProps` / `getStaticProps`） | `app` 下默认 RSC，数据获取更常写在 async Server Component / `fetch` 缓存语义里 |
  | 「服务端渲染」直觉来源 | `getServerSideProps` = 请求时；`getStaticProps` = 构建时 | Dynamic vs Prerender + `generateStaticParams` 等文件约定 |
  | 导航体验关键词 | `<Link>` prefetch 以页面为单位（随版本有演进） | 文档强调 payload + prefetch + streaming + transition 组合 |

2. Prefetching
   next/link 的 \<Link\> 会在链接进入视口时自动预取目标路由；
   - 静态路由往往预取整段路由;
   - 动态路由可能不预取或只做部分预取，若有 loading.tsx 则部分预取（例如共享布局Layout + loading UI），避免用户可能永远不去的动态页在服务端被白做一堆工作

   **Pages Router 对照（Prefetching）**

   | 维度         | Pages Router                                                                                    | App Router                                                                   |
   | ------------ | ----------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------- |
   | 组件         | 也是 `next/link`                                                                                | 也是 `next/link`                                                             |
   | 预取对象形态 | 更偏传统「页面数据 / 路由 JSON」那套历史模型                                                    | 更偏 RSC payload / 流式片段（与 Layout、`loading.tsx`、Suspense 边界强相关） |
   | 动态路由体验 | 也有 prefetch 行为与限制，但「loading 文件约定 + 部分预取」在 App Router 里由文件约定讲得更直接 | 官方文档常把 `loading.tsx` 与动态路由的 partial prefetch 放在一起说明        |

3. Streaming
   流式渲染：服务器不必等整条路由全部算完，就可以先把已经就绪的部分（例如共享 layout、loading.tsx 里的占位 UI）发给浏览器；慢的那块（例如 page 里的异步数据）稍后继续流式补上.

   > 因为 动态路由 若只靠「整页等服务器」，用户会觉得卡住；加上 loading.tsx 后，Next 会做 partial prefetch（部分预取）， \<Link\> 导航可以更快给出反馈

   > | 维度                                  | App Router                                                                     | Pages Router                                                                                                                                |
   > | ------------------------------------- | ------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------- |
   > | 文件约定                              | `app/.../loading.tsx` 由框架挂成该段的 Suspense fallback                       | 无 `loading.tsx` 同名约定；需在组件树里自行包 `<Suspense>`，否则多为「整段就绪再一次性展示」                                                |
   > | 与 RSC / 数据模型                     | 默认 RSC，流式与 Flight payload 一致，和 `loading.tsx`、嵌套 layout 同一套叙事 | 以 `pages/` + `getServerSideProps` / `getStaticProps` 等为主；是否流式、边界在哪，与 Next/React 版本及写法强相关，和 App 不是同一套文件约定 |
   > | 文档里的 partial prefetch + `loading` | 动态段 + `loading.tsx` 是官方主推组合                                          | 无同一套「部分预取 + loading 文件」说明；对照时抓住本质：**分段就绪**减轻过渡卡顿，而不是照搬文件名                                         |
