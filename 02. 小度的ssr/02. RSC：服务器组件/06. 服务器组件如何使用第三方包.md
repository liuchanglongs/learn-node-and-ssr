### 服务器组件的处理

把第三方包提取成一个单独的组件，并添加"use client"

### 有利于RSC

- 原因（对应到 Next.js）

| 维度          | 整页 `07.tsx` + `"use client"`（第一种）                                               | 页面默认 Server + `<SliderImage />`（第二种）                  |
| :------------ | :------------------------------------------------------------------------------------- | :------------------------------------------------------------- |
| **RSC 边界**  | 该文件（及默认子树）整段都是 Client，页面无法再当 Server Component 用                  | 页面仍是 **Server Component**，只有 `SliderImage` 一块进客户端 |
| **数据获取**  | 以后若在页面里写 `async`、读 DB、用 `cookies()` 等，会和 `"use client"` 冲突，只能再拆 | 页面可以继续 **服务端取数**，把结果当 props 传给客户端轮播     |
| **客户端 JS** | 容易把整个页面的逻辑都绑在 client bundle 上                                            | **Client 边界更小**，更符合「能服务端就服务端」的习惯          |
| **可读性**    | 小 demo 可以，一复杂就臃肿                                                             | 页面负责结构/数据，交互 UI 单独文件，更清晰                    |

### Pages Router 对照（若你也在学旧路由）

- **Pages Router**：`pages/xxx.tsx` 没有 RSC，整页本质都是「先服务端出 HTML + 再 hydrate」，没有「默认 Server Component 页面」这一说；抽不抽组件主要是 **代码组织**，对「Server vs Client 边界」影响不如 App Router 大。
- **App Router**：第二种写法 **明显更贴合 RSC 设计**。
