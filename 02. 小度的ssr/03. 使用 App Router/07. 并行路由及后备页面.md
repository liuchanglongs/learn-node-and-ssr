### 并行路由： @槽位名

- App Router 里用 @槽位名 文件夹 把「`同一 URL `下要并排渲染的多块 UI」拆成 多个独立子树：每块可以有自己的 page.tsx、loading.tsx、error.tsx 等，由 父级 layout.tsx 通过同名 props 同时接住并排版（solot概念：children是默认的->对应的page）。
- 当一个页面功能比较复杂的情况下，可以拆解成多个组件。但是这种方式是没有自己单独的loading、error等行为的，所以并行路由可以更好的处理这种复杂的需求。

- 并行路由允许你同时或有条件地渲染同一布局中的一个或多个页面。它们对于应用的高度动态部分非常有用，例如仪表板和社交网站上的提要(信息密度高、多块区域独立更新、每块都想有自己的 loading / 错误边界 / 数据获取)。

### 后备页面

- 当前这次导航解析下来，**某个并行槽（@xxx）在「它该负责的那一段 URL」下，匹配不到对应的 page（等有效页面）时，就用这个槽（或该段）的 default.tsx 来填这个槽位**。

1. 在 软导航 期间，Next.js 跟踪每个槽的活动状态（子页面）。
   > 用 Link / router.push 等在 **不整页刷新(csr一样，除非有状态干扰)** 的前提下换 URL 时，Next 在客户端还保留着应用状态，能记住每个并行槽（@card / @menu 等）之前激活的是哪一段子路由，从而在「新 URL 并没有给某个槽明确指定页面」时，仍有机会维持合理的 UI（ 「跟踪每个槽的活动状态」）。
2. 但是，对于硬导航（全页加载），Next.js 无法恢复活动状态。在这种情况下，可以为与当前 URL 不匹配的子页面渲染 default.tsx 文件。

> 地址栏回车、刷新、外链直达、新开 Tab 等 整页重新加载 时，浏览器里 没有 上一轮 SPA 的内存状态，Next 无法凭空知道「你上次在 @menu 里看的是哪一页」，只能严格按 当前 URL 能匹配到的路由文件 来渲染。若某个槽在当前 URL 下 没有对应的 page.tsx，就必须靠该槽的 default.tsx（后备） 告诉 Next：这个槽位显示什么，否则容易出现空白、不匹配或构建/运行时的路由不完整问题（视具体版本与结构而定）。

### 理解并行路由、后备页面

1. 每个 @槽位 是独立子树
   URL 里能映射到 @card/...、@menu/... 的 segment 各自去匹配各自的 page.tsx。

2. 软导航 = 换 URL + 客户端取新 RSC payload
   新 URL 下，某个槽若从「匹配 A」变成「匹配 B」，**这个槽的 React 子树就会按新路由重渲染，看起来就是 槽位里的内容换了**。

3. 不是每个槽每次都会换
   若新 URL 只改了另一个槽 对应的 segment，其它槽的匹配结果没变，那些槽可以 保持原样（不必跟着「一起替换」）。

4. 若没有匹配到该槽的 page
   会落到该槽的 default.tsx（硬导航时尤其依赖这个）；软导航时官方还会提到「记住槽的活动状态」，但 可依赖的兜底仍是文件系统里的 default，不要指望靠内存「猜」出未写在 URL 里的槽内容。
